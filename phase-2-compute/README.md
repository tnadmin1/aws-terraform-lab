# Phase 2 — EC2 Compute, Key Pairs, and IAM

Deploys compute resources into the Phase 1 VPC: a bastion host in a public subnet, a private instance accessible only through the bastion, a generated SSH key pair, and an IAM role with least-privilege policies attached via an instance profile.

---

## Resources Deployed

| Resource | Name | Description |
|----------|------|-------------|
| `aws_instance` | tf-lab-bastion | Bastion host in public subnet — SSH entry point |
| `aws_instance` | tf-lab-private-instance | Private instance — no public IP, reachable via bastion or SSM |
| `tls_private_key` | — | RSA 4096-bit key generated by Terraform |
| `aws_key_pair` | tf-lab-bastion-key | Public key registered in AWS |
| `local_file` | tf-lab-bastion.pem | Private key saved locally with chmod 400 |
| `aws_iam_role` | tf-lab-ec2-role | EC2 assume-role with SSM + CloudWatch policies |
| `aws_iam_instance_profile` | tf-lab-ec2-profile | Wraps the IAM role for EC2 attachment |
| `aws_security_group` | tf-lab-bastion-sg | SSH inbound from allowed CIDRs only |
| `aws_security_group` | tf-lab-private-instance-sg | SSH inbound from bastion SG only — not the internet |

---

## Prerequisites

Phase 1 must be deployed and the `tf-lab-vpc` VPC must exist with `Tier=public` and `Tier=private` subnet tags applied. Phase 2 uses data sources to discover these resources dynamically — no hardcoded IDs.

---

## Usage

```bash
cd phase-2-compute

# Add the hashicorp/tls provider (needed for key generation)
terraform init

terraform plan
terraform apply
```

After apply, retrieve sensitive outputs:

```bash
# Get SSH command
terraform output -raw ssh_command

# Get private key path
terraform output -raw private_key_path
```

---

## Connecting to Instances

**Bastion host (direct SSH):**
```bash
ssh -i tf-lab-bastion.pem ec2-user@<bastion_public_ip>
```

**Private instance (via bastion jump):**
```bash
ssh -i tf-lab-bastion.pem -J ec2-user@<bastion_public_ip> ec2-user@<private_ip>
```

**Private instance (via SSM — no SSH required):**
```bash
aws ssm start-session --target <private_instance_id>
```

## Design Decisions

- **Data sources over hardcoded IDs** — Phase 2 queries AWS for the VPC and subnets by tag, making it independent of Phase 1's state file. This is the pattern used in real multi-team environments.
- **Terraform-generated key pair** — the `tls` provider generates the key so everything is reproducible and tracked. The private key is marked sensitive and saved locally with correct permissions.
- **SSM over open SSH** — the IAM role includes `AmazonSSMManagedInstanceCore`, enabling AWS Systems Manager Session Manager as an alternative to SSH. This is the modern production pattern — no port 22 required.
- **Security group chaining** — the private instance SG references the bastion SG as its ingress source rather than a CIDR. This means only traffic originating from the bastion is allowed, regardless of IP changes.
- **Encrypted EBS volumes** — both instances use `encrypted = true` on root volumes. Default in production; explicitly set here for visibility.
- **Amazon Linux 2023** — selected via `data.aws_ami` with `most_recent = true` so the AMI ID stays current without manual updates.
